# Настройка своего бесплатного Firebase-проекта

Этот проект умеет хранить профили расположения мест не только в
`localStorage` браузера, но и в облаке — на бесплатном тарифе Firebase
(Spark plan). Тогда профили, сохранённые в `admin.html`, сразу видны всем
посетителям `index.html`, без необходимости передавать файл вручную.

Ниже — пошаговая инструкция, как завести **свой собственный** Firebase-проект
и подключить его к этому сайту. Повторять придётся один раз, дальше всё
работает само.

Если облако не нужно — можно ничего не делать: сайт полностью работает и
без Firebase, просто каждый профиль будет сохраняться только в
браузере того, кто его создал (в `localStorage`).

## Что понадобится

- Google-аккаунт (обычная почта Gmail подойдёт).
- 10–15 минут свободного времени.
- Бесплатный тариф Firebase (Spark) — банковская карта не требуется.

## Шаг 1. Создать проект в Firebase Console

1. Откройте <https://console.firebase.google.com/> и войдите под своим
   Google-аккаунтом.
2. Нажмите **Add project / Создать проект**.
3. Введите название проекта (например, `bus-seat-planner`) и нажмите
   **Continue**.
4. На вопрос про Google Analytics можно ответить «нет» (для этого проекта
   не нужен) — переключатель **Enable Google Analytics** можно выключить.
5. Нажмите **Create project** и подождите, пока Firebase создаст проект.

## Шаг 2. Зарегистрировать веб-приложение и получить конфиг

1. На главной странице проекта нажмите иконку **`</>`** ("Web") —
   добавить веб-приложение.
2. Введите любое название приложения (например, `bus-seat-planner-web`).
   Галку **Firebase Hosting** можно не ставить — этот проект публикуется
   через GitHub Pages, хостинг Firebase не используется.
3. Нажмите **Register app**.
4. Firebase покажет объект `firebaseConfig` — что-то вроде:

   ```js
   const firebaseConfig = {
     apiKey: "AIzaSy...",
     authDomain: "ваш-проект.firebaseapp.com",
     projectId: "ваш-проект",
     storageBucket: "ваш-проект.firebasestorage.app",
     messagingSenderId: "1234567890",
     appId: "1:1234567890:web:abcdef1234567890",
   };
   ```

   Скопируйте эти значения — они понадобятся в шаге 5. Это **не секретный
   ключ**: `apiKey` в веб-конфиге Firebase рассчитан на то, чтобы быть
   публичным (доступ на самом деле контролируется правилами безопасности
   Firestore, см. шаг 4), поэтому его можно спокойно закоммитить в
   публичный репозиторий.

5. Нажмите **Continue to console**.

## Шаг 3. Включить Firestore (базу данных)

1. В левом меню откройте **Build → Firestore Database** (может называться
   просто **Firestore**, в интерфейсе на русском — «База данных»).
2. Нажмите **Create database**.
3. Выберите режим **Start in production mode** (правила допишем сами на
   шаге 4) и нажмите **Next**.
4. Выберите регион (любой ближайший к вашим пользователям, например
   `eur3 (europe-west)`) и нажмите **Enable**.

## Шаг 4. Настроить правила безопасности Firestore

По умолчанию Firestore в режиме production запрещает вообще любой доступ.
Нужно явно разрешить: **чтение — всем**, **запись — только вошедшим
администраторам**.

1. В Firestore откройте вкладку **Rules**.
2. Замените содержимое на:

   ```
   rules_version = '2';
   service cloud.firestore {
     match /databases/{database}/documents {
       match /profiles/{profileId} {
         allow get, list: if true;
         allow create, update, delete: if request.auth != null;
       }
     }
   }
   ```

   Готовая копия этих правил лежит в файле [`firestore.rules`](firestore.rules)
   в этом репозитории — можно скопировать оттуда.

   > **Важно:** `allow get, list: if true` — это два разных права
   > (`get` — чтение одного документа, `list` — чтение коллекции целиком).
   > Если указать только `allow read: if true`, часть запросов (`list`,
   > то есть подписка на весь список профилей) всё равно будет
   > отклоняться с ошибкой 403 «Missing or insufficient permissions» —
   > нужно перечислить оба права явно, как показано выше.

3. Нажмите **Publish**.

## Шаг 5. Включить вход по email/паролю и создать администратора

1. В левом меню откройте **Build → Authentication**.
2. Нажмите **Get started**.
3. На вкладке **Sign-in method** включите провайдер **Email/Password**
   (нажмите на него, переключите **Enable**, нажмите **Save**).
4. Перейдите на вкладку **Users** и нажмите **Add user**.
5. Введите email и пароль администратора — под этими данными вы будете
   заходить в `admin.html`, чтобы сохранять и удалять профили. Нажмите
   **Add user**.

   > Держите этот пароль в секрете и не публикуйте его в чатах/коде —
   > это единственная защита от того, чтобы посторонние могли менять
   > профили на вашем сайте. Поменять пароль позже можно на той же
   > вкладке **Users**.

## Шаг 6. Вставить свой конфиг в проект

1. Откройте файл [`firebase-config.js`](firebase-config.js) в корне
   репозитория.
2. Замените значения на те, что вы скопировали в шаге 2:

   ```js
   window.FIREBASE_CONFIG = {
     apiKey: "...",
     authDomain: "...",
     projectId: "...",
     storageBucket: "...",
     messagingSenderId: "...",
     appId: "...",
   };
   ```

3. Сохраните файл и опубликуйте изменения (закоммитьте и запушьте, если
   используете Git, либо просто загрузите файл на хостинг).

Больше никаких изменений в коде не требуется — `cloudSync.js` сам подхватит
конфиг из `firebase-config.js` при загрузке страницы.

## Шаг 7. Проверить, что всё работает

1. Откройте `admin.html` на вашем сайте. Сверху должен появиться зелёный
   блок «Облачное хранилище» с полями email/пароль.
2. Введите email и пароль администратора (из шага 5) и нажмите **Войти**.
   Статус должен поменяться на «вы вошли как ...».
3. Создайте или отредактируйте любой профиль и нажмите **Сохранить
   профиль**.
4. Откройте `index.html` (можно в другом браузере или в режиме
   инкогнито, без входа) — сохранённый профиль должен появиться в списке
   выбора профилей практически сразу, без обновления страницы (обновление
   происходит в реальном времени через Firestore `onSnapshot`).

Если профиль не появляется — проверьте:

- Что в консоли браузера (F12 → Console) нет ошибок вида `Missing or
insufficient permissions` (значит, не опубликованы правила из шага 4)
  или `auth/invalid-api-key` (значит, неверно скопирован конфиг в шаге 6).
- Что в Firebase Console на вкладке **Firestore → Data** появилась
  коллекция `profiles` с вашим сохранённым профилем.

## Бесплатный лимит (тариф Spark)

Тариф Spark (бесплатный) даёт, в частности:

- Firestore: 1 ГиБ хранилища, 50 000 чтений и 20 000 записей в день.
- Authentication: неограниченное количество пользователей с
  email/password.

Для схемы мест в автобусе (несколько десятков небольших профилей и
редкие сохранения администратором) этого более чем достаточно — сайт с
обычной посещаемостью экскурсионного/трансферного сервиса не должен
приближаться к этим лимитам. Актуальные цифры лучше проверять на
странице <https://firebase.google.com/pricing>, так как Google может их
менять.

## Ограничения для пользователей из России

На момент написания этой инструкции Firebase (как и большинство сервисов
Google Cloud) не имеет специальных юридических блокировок по признаку
гражданства/местоположения в России для использования публичных
клиентских SDK, подобных используемому в этом проекте. Тем не менее:

- Доступ к консоли Firebase и самим сервисам Google может зависеть от
  текущей сетевой обстановки (блокировки/ограничения на стороне
  провайдеров), а не только от политики Google.
- Оплата платных тарифов Firebase с российских банковских карт может
  быть недоступна из-за санкционных ограничений платёжных систем — но
  бесплатный тариф Spark, описанный в этой инструкции, не требует
  привязки карты вовсе.
- Точную и актуальную информацию по ограничениям стоит проверять
  непосредственно перед использованием, так как ситуация может
  меняться.
